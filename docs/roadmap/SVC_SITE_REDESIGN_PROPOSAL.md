# SALT Verification Console 사이트 개편안 (v1.0)

최종 업데이트: `2026-03-06`

## 1) 개편 목적
- 방문자가 10초 안에 다음을 이해하도록 설계한다.
  1. 이 사이트가 무엇을 주장하는가
  2. 어떤 데이터로 검증하는가
  3. 표준이론 대비 SALT가 어디서 더 정확한가
- 결론: “DB 나열”이 아니라 “검증 스토리 + 근거 시각화” 중심으로 재구성한다.

## 2) 핵심 메시지(고정 문구)
- 메인 메시지:  
  `SALT는 특정 관측 구간에서 표준이론보다 낮은 예측 오차를 보입니다.`
- 보조 메시지:  
  `모든 결론은 공개 데이터와 사전고정 지표(MAE/RMSE/Residual)로만 제시합니다.`
- 주의 메시지:  
  `SALT 우위는 전체가 아니라, 데이터와 구간별로 판정됩니다.`

## 3) 정보구조(IA) 개편
현재 다중 메뉴를 아래 5개로 단순화한다.

1. `Home (결론 요약)`
2. `Evidence (핵심 비교 시각화)`
3. `Events (사건별 상세)`
4. `Method (데이터/평가 규칙)`
5. `Limits (한계/반증/실패사례)`

비활성/후순위:
- `Book`, `Roadmap`은 푸터 링크로 이동
- 일반 방문자 1차 동선에서 제외

## 4) 페이지별 의도와 화면 구성

### A. Home
- 의도: 사이트 목적을 한 문장으로 전달
- 구성:
  - Hero: “표준 vs SALT, 실제 오차 비교”
  - KPI 카드 3개:
    - `전체 이벤트 수`
    - `SALT 우세 구간 비율`
    - `최근 업데이트 시각`
  - “지금 보기” 버튼 -> `Evidence`

### B. Evidence (핵심 페이지)
- 의도: 방문자가 SALT 우위를 한눈에 확인
- 구성:
  - 그래프 1: `Actual vs Standard vs SALT` 시계열 3중 선그래프
  - 그래프 2: `Residual(Standard)` vs `Residual(SALT)` 비교 막대/라인
  - 그래프 3: 윈도우별(`1h/24h/7d`) MAE/RMSE 카드
  - 필터: `데이터셋`, `이벤트 타입`, `기간`, `신뢰도`
  - 자동 설명 박스:
    - “이 필터 조건에서 SALT가 표준 대비 RMSE xx% 감소”

### C. Events
- 의도: “전체 통계”가 아닌 개별 사례 검증
- 구성:
  - 이벤트 리스트(최신순)
  - 이벤트 상세:
    - 원 데이터 요약
    - 표준 예측값/오차
    - SALT 예측값/오차
    - 판정(`SALT 우세 | 동등 | 표준 우세`)

### D. Method
- 의도: 신뢰 확보
- 구성:
  - 데이터 출처(GWOSC/GraceDB/GCN/ZTF/HEASARC)
  - 지표 정의(MAE/RMSE/Residual/DeltaFit)
  - 블라인드/반증 규칙
  - 재현 절차(스크립트 실행 순서)

### E. Limits
- 의도: 과장 방지 및 반론 대응
- 구성:
  - SALT 실패 사례
  - 표준이 더 잘 맞는 구간
  - 데이터 결측/품질 이슈
  - 현재 미해결 항목

## 5) 데이터 모델 재정렬(표시 중심)
DB는 유지하되, 프론트 노출 모델은 아래 3축으로 고정한다.

1. `actual_value` (실측)
2. `standard_pred` (표준이론 예측)
3. `salt_pred` (SALT 예측)

파생값:
- `error_standard = actual - standard_pred`
- `error_salt = actual - salt_pred`
- `winner = argmin(|error_standard|, |error_salt|)`

## 6) 시각화 원칙
- 원칙 1: 항상 `실측`을 함께 표시 (예측끼리만 비교 금지)
- 원칙 2: 오차축은 절대값/부호를 분리 표기
- 원칙 3: 결론은 문장보다 수치 우선
- 원칙 4: 실패 사례를 기본 탭으로 제공

## 7) UX 카피 가이드
- 나쁜 문구: “SALT가 절대적으로 우월하다”
- 권장 문구: “이 데이터/이 기간/이 조건에서 SALT 오차가 더 낮다”
- 숫자 표기:
  - `RMSE 0.42 -> 0.31 (26.2% 감소)`
  - `n=184 events`

## 8) 구현 단계 (실행 순서)
1. Phase 1: 내비게이션 단순화 + Home/Evidence 우선 배치
2. Phase 2: 3중 비교 그래프(`actual/standard/salt`) 구현
3. Phase 3: Events 상세 판정 화면 구축
4. Phase 4: Limits 페이지(실패사례) 추가
5. Phase 5: Method 정리/자동 리포트 연결

## 9) 완료 기준(Definition of Done)
- 방문자 5명 테스트에서 “사이트 목적” 정답률 80% 이상
- Evidence 페이지에서 30초 내 핵심 그래프 도달 가능
- 모든 주장에 대응하는 수치/소스 링크 존재
- 실패 사례 최소 1개 이상 상시 노출

## 10) 즉시 착수 항목

- [ ] Evidence 페이지 그래프 컴포넌트 1차 구현
- [ ] 이벤트 판정 규칙(`winner`) 프론트 표기 추가
