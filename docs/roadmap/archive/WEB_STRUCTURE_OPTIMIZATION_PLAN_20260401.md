# Web Structure Optimization Plan

기준일: `2026-04-01`

작성 목적: `docs/book/canonical_SALT_Logic_System_ERD.md`의 논리 구조를 기준으로, 현재 웹 정보구조가 얼마나 적절한지 평가하고 다음 개편 순서를 정한다.

## 1. 최적화의 정의

이 사이트에서 구조 최적화는 한 가지 기준만으로 판단할 수 없다. 아래 다섯 축을 동시에 만족해야 한다.

### 1) 기능적 효율성

- 방문자가 최소한의 클릭으로 `도서 소개`, `핵심 개념`, `검증 결과`, `참고 자료`, `재현 경로`에 도달할 수 있어야 한다.
- 운영자 입장에서는 라우트와 콘텐츠 구조가 중복 없이 유지되어야 한다.

### 2) 가독성

- 한 페이지가 한 가지 역할을 가져야 한다.
- 개념, 결과, 가설, 감사 자료가 한 화면에서 뒤섞이지 않아야 한다.

### 3) 맥락 전달

- 방문자는 먼저 `이 사이트가 책의 어떤 부분을 어떻게 안내하는지` 이해할 수 있어야 한다.
- 개별 주장보다 `읽기 흐름`이 먼저 보이는 구조가 유리하다.

### 4) 방문자 관리

- 처음 온 독자, 구매 의사가 있는 방문자, 검증만 보려는 검토자, 기술적 관심 방문자의 경로가 달라야 한다.
- 구조는 이 경로를 강요하기보다 자연스럽게 분기시켜야 한다.

### 5) frontend-backend 조화

- 메뉴 구조, 라우트 구조, 페이지 맵, 데이터 모델이 서로 같은 논리를 따라야 한다.
- 화면 구조와 코드 구조가 어긋나면 유지보수 비용이 급격히 올라간다.

## 2. 판단 기준

위 기준을 바탕으로, 이 사이트에서 가장 적절한 구조는 다음처럼 정의할 수 있다.

> 책의 논리 구조를 그대로 복제하는 구조가 아니라,
> 방문자가 책의 맥락을 가장 안정적으로 이해하도록
> 논리 구조를 웹 동선으로 번역한 구조

따라서 ERD는 `논리 원형`, 웹 IA는 `방문자용 번역`, 페이지 맵과 콘텐츠 모델은 `두 층을 연결하는 중간층`으로 보는 것이 맞다.

## 3. 현재 구조 평가

### 3.1 잘 맞는 점

- 상위 허브가 `Guide / Core / Verification / Engineering / Reference / Audit`로 분리되어 있어 방문자 동선이 명확하다.
- `Verification`과 `Engineering`이 분리되어 있어 검증 결과와 기술적 해석이 구분된다.
- `Reference`와 `Audit`가 따로 있어 참고 자료와 재현 자료의 역할이 섞이지 않는다.
- 홈과 Guide에서 도서 소개 사이트로서의 진입 동선이 비교적 안정적이다.

### 3.2 부족한 점

- ERD의 `Problem -> Clue -> Concept -> Solution -> Verify` 흐름이 웹의 명시적 구조로는 드러나지 않는다.
- `Core`가 사실상 `17/18/19장 압축 허브`로 작동하고 있어, ERD 중심의 개념 지도 역할은 아직 약하다.
- `page-map.ts`가 현재 실제 라우트 구조를 충분히 반영하지 못한다.
- 개별 페이지의 설명 데이터가 컴포넌트 내부 상수에 남아 있어 구조적 일관성이 약하다.
- 책 장 구조와 웹 구조의 대응은 [reference/book-map/page.tsx](/home/jwater/Development/salt-verification-console/web/src/app/reference/book-map/page.tsx)에만 부분적으로 드러나고, 전체 네비게이션에서는 충분히 활용되지 않는다.

## 4. 핵심 판단

현재 웹 구조는 `방문자용 정보구조`로는 상당히 적절하지만, `ERD를 가장 잘 번역한 최적 구조`라고 보기는 어렵다.

정리하면 현재 상태는 다음과 같다.

- 방문자 동선: `양호`
- 톤과 역할 분리: `양호`
- ERD 대응성: `부분적`
- 페이지 맵 및 데이터 모델 정합성: `미완료`

즉, 구조를 전면 폐기할 필요는 없지만, `ERD-웹-데이터 모델`을 다시 한 번 맞추는 2차 정리가 필요하다.

## 5. 목표 구조

다음 단계에서는 현재 상위 IA를 유지하되, 내부 연결 논리를 더 명확히 만드는 것이 좋다.

### A. 방문자용 최상위 구조

- `Home`
- `Guide`
- `Core`
- `Verification`
- `Engineering`
- `Reference`
- `Audit`

이 구조는 유지한다.

### B. ERD 대응 중간층

`Core` 내부에 아래 축을 명시적으로 드러내는 것이 바람직하다.

- `Problem`
- `Clue`
- `Concept`
- `Solution`
- `Bridge to Verification`

이 중간층은 별도 설명 페이지이거나 `logic map` 페이지로 구현하는 것이 적절하다.

### C. 데이터/페이지 맵 계층

아래 공통 모델이 필요하다.

- `site_section`
- `book_chapter_summary`
- `erd_stage`
- `verification_channel`
- `reference_asset`

각 페이지는 이 공통 모델을 불러와 화면에 렌더링하는 식으로 정리하는 것이 유지보수에 유리하다.

## 6. 실행계획

### Phase 1. 구조 기준 고정

우선순위: `P0`

할 일:

- ERD 단계명을 웹 내부 구조에서 어떻게 드러낼지 결정
- `Core`의 역할을 `장 요약 허브`인지 `논리 구조 허브`인지 명확히 고정
- `Verification`, `Engineering`, `Reference`, `Audit`의 경계 문장을 짧게 표준화

산출물:

- 구조 기준 문서 1개
- 상위 섹션 정의표 1개

### Phase 2. 빠진 라우트 마감

우선순위: `P0`

할 일:

- `/core/logic-map` 추가
- `/core/chapters` 인덱스 추가
- `/verification/channels` 인덱스 추가
- `/verification/candidate-hypotheses` 추가 여부 결정 및 반영
- `site-structure-map.tsx`와 실제 라우트의 불일치 제거

산출물:

- 누락 라우트 페이지
- 상단 드롭다운과 실제 IA 일치

### Phase 3. 페이지 맵과 콘텐츠 모델 정리

우선순위: `P1`

할 일:

- [page-map.ts](/home/jwater/Development/salt-verification-console/web/src/lib/page-map.ts) 전면 동기화
- 페이지 내부 상수 배열을 `lib/data` 또는 `content` 계층으로 이동
- 장, ERD 단계, 웹 섹션의 대응 테이블 작성

산출물:

- 정리된 페이지 맵
- 재사용 가능한 콘텐츠 데이터 계층

### Phase 4. 방문자 동선 미세조정

우선순위: `P1`

할 일:

- 홈과 Guide에서 `처음 온 독자`, `구매 의사 방문자`, `검증 검토자`의 경로를 더 명확히 분리
- `Book Map`과 `Core Logic Map`의 역할을 구분
- `Reference`가 단순 보조 허브가 아니라 `책-웹 대응 허브`로 보이도록 보강

산출물:

- CTA 정리
- Book Map / Logic Map 역할 분리

### Phase 5. 유지보수 구조 확정

우선순위: `P2`

할 일:

- ERD 문서 갱신 시 웹 구조에 어떤 항목이 영향을 받는지 규칙화
- 원고 변경과 웹 요약 변경의 동기화 지점 정의
- 이후 댓글/Auth/DB 확장 시 어느 섹션에 연결할지 미리 정리

산출물:

- 변경 영향도 규칙
- 유지보수 가이드

## 7. 권장 우선순위

지금 바로 손대야 하는 순서는 아래와 같다.

1. `Core`를 ERD 중심 허브로 보강할지 여부 결정
2. `/core/logic-map`과 `/core/chapters` 추가
3. `/verification/channels` 인덱스와 후보 가설 분리
4. `page-map.ts`와 드롭다운 구조 동기화
5. 페이지 내부 상수 데이터 분리

## 8. 결론

현재 구조는 `완전히 잘못된 구조`는 아니다. 오히려 방문자 관점에서는 이미 상당히 안정적이다.

다만 아직 최적 구조라고 말하려면 아래 두 조건이 더 필요하다.

- ERD 논리 구조가 `Core`와 `Book Map`을 통해 더 명시적으로 드러날 것
- 페이지 맵과 콘텐츠 데이터 구조가 실제 화면 구조와 완전히 일치할 것

따라서 다음 개편은 `전면 재설계`보다 `논리 중간층 보강 + 데이터 구조 정렬` 쪽으로 가는 것이 맞다.
