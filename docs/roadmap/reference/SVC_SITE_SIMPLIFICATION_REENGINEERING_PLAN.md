# SVC Site Simplification Reengineering Plan

기준일: `2026-04-05`

이 문서는 `salt.numverse.org`를 더 단순하고 직관적인 입문형 웹사이트로 재구성하기 위한 별도 설계안이다.

주의:

- 현재 실행의 최종 기준 문서는 [MASTER_EXECUTION_STRUCTURE_PLAN.md](/home/jwater/Development/salt-verification-console/docs/roadmap/current/MASTER_EXECUTION_STRUCTURE_PLAN.md)다.
- 이 문서는 그 마스터 문서를 대체하지 않고, `사이트 단순화`라는 특정 목적에 집중한 리엔지니어링 제안서다.

## 1. 문제 정의

현재 사이트는 정보가 많고 구조도 비교적 잘 잡혀 있다. 그러나 처음 온 독자 입장에서는 아래 문제가 있다.

- `SALT가 무엇인지`보다 `사이트 구조`가 먼저 보인다.
- 전역 내비, 허브 페이지, 하위 링크가 겹쳐 보여 정보가 과밀하게 느껴진다.
- 검증, 감사, provenance, snapshot 같은 고급 개념이 너무 앞단에 노출된다.
- 독자는 `그래서 SALT가 대체 무슨 아이디어인가`를 빠르게 붙잡기 어렵다.

결과적으로 현재 웹은 `설명 가능한 모든 것`을 담으려는 구조에 가깝고, `처음 이해시키는 구조`로는 다소 무겁다.

## 2. 재정의할 사이트 역할

웹사이트의 1차 역할은 아래 하나로 축소한다.

- `SALT의 핵심 아이디어를 처음 온 독자도 직관적으로 이해할 수 있게 소개하고, 더 깊은 논증은 책과 검증 자료로 안내한다.`

즉 역할 우선순위는 아래처럼 다시 정리한다.

1. 입문 안내
2. 핵심 인사이트 제시
3. 책으로 유도
4. 검증 근거 연결
5. 감사/재현 자료 제공

핵심은 `웹이 책의 축약본 전체가 되려고 하지 않는다`는 점이다.

## 3. 리엔지니어링 원칙

### 3.1 내용보다 구조가 먼저 보이지 않게 한다

- 구조 안내는 전역 내비가 담당한다.
- 본문은 항상 `메시지 -> 설명 -> 다음 이동` 순서를 따른다.
- 링크 허브, 분기 카드, 자료실 느낌의 블록은 첫 화면에서 최소화한다.

### 3.2 한 페이지는 한 메시지만 가진다

- 한 페이지에서 `소개`, `검증`, `감사`, `참고 자료`를 동시에 설명하지 않는다.
- 각 페이지는 첫 문장만 읽어도 무슨 페이지인지 이해되어야 한다.

### 3.3 숫자보다 해석 문장이 먼저 온다

- 퍼센트, 우세 수치, 버전명은 단독으로 두지 않는다.
- 항상 `무엇을 비교했는지`, `몇 개 중 몇 개인지`, `왜 봐야 하는지`를 함께 쓴다.

### 3.4 고급 자료는 2층 이하로 내린다

- `runs`, `snapshots`, `manifest`, `hash`, `formula version`은 공개 유지
- 단, 첫 진입 화면에서는 숨기고 `더 확인하기` 아래에 둔다.

### 3.5 복잡한 논증은 책으로 보낸다

- 웹은 핵심 문제의식과 인사이트만 전달한다.
- 장문의 증명, 부록형 설명, 세부 프로토콜은 책 또는 감사 라우트로 넘긴다.

## 4. 목표 정보구조

현재 최상위 구조를 방문자 노출 기준으로 더 단순하게 읽히도록 재해석한다.

### 4.1 방문자 노출 기준 상위 구조

- `소개`
- `핵심 아이디어`
- `검증과 근거`
- `책과 참고`

실제 내부 라우트는 지금 구조를 유지할 수 있다. 하지만 방문자에게는 위 4개의 큰 의미만 보이게 만든다.

### 4.2 내부 구조와의 대응

| 방문자용 상위 개념 | 내부 라우트 대응 | 역할 |
| --- | --- | --- |
| `소개` | `/`, `/guide` | 처음 온 독자에게 SALT의 문제의식과 읽는 법을 소개 |
| `핵심 아이디어` | `/core`, `/core/logic-map`, 일부 `/engineering` | SALT가 무엇을 새롭게 제안하는지 직관적으로 설명 |
| `검증과 근거` | `/verification`, `/verification/results`, `/snapshots`, `/runs` | 실제 비교 결과와 근거 자료의 위치 안내 |
| `책과 참고` | `/reference`, `/reference/book-map`, 구매 링크 | 더 깊은 논증과 책 독서를 위한 연결 |

### 4.3 Audit의 위치 재정의

- `Audit`는 전면 허브가 아니라 `검증과 근거` 아래의 고급 영역으로 재배치한다.
- 일반 독자에게는 `근거 더 보기` 정도로 노출하고, 고급 사용자가 들어가면 provenance를 본다.

## 5. 추천 라우트 전략

### 5.1 바로 바꾸기보다 “보이는 의미”를 바꾼다

라우트명 전체를 한 번에 갈아엎기보다, 먼저 페이지 역할과 내비 표현을 정리한다.

1단계에서는 다음을 우선 적용한다.

- 상단 내비 문구를 입문형으로 재정렬
- 홈/Guide/Core/Verification 첫 문장을 재작성
- 페이지 첫 화면에서 링크 카드 비중 축소
- `Audit`와 `Reference`를 전면 노출에서 한 단계 후순위로 내림

### 5.2 장기적으로 고려할 단순화 라벨

필요하면 향후 아래처럼 라벨 재정리도 가능하다.

- `Guide` -> `소개`
- `Core` -> `핵심 아이디어`
- `Verification` -> `검증과 근거`
- `Reference` -> `책과 참고`
- `Audit` -> `고급 감사 자료`

단, 라우트 자체를 바꾸는 것은 후순위다. 지금은 `독자가 보는 언어`를 먼저 바꾸는 것이 맞다.

## 6. 페이지별 리엔지니어링 설계

### 6.1 Home

Home은 사이트 구조 안내판이 아니라 `SALT 한 줄 소개 + 세 가지 핵심 인사이트` 페이지가 되어야 한다.

남길 것:

- SALT가 무엇인지 한 줄 정의
- 왜 흥미로운지 세 문장
- 책 구매/책 보기 CTA
- 검증 결과 보기 CTA 하나

줄일 것:

- 과도한 섹션 소개 카드
- 첫 화면에서의 provenance 설명
- 너무 많은 허브 링크

추천 블록 순서:

1. 한 줄 정의
2. SALT의 핵심 인사이트 3개
3. 왜 중요한가
4. 지금 확인할 수 있는 것
5. 책 보기 / 검증 보기

### 6.2 Guide

Guide는 구조 설명 전용으로 유지하되 더 짧아져야 한다.

목표:

- `처음 온 독자`
- `책을 사려는 독자`
- `검증만 보려는 사람`
- `기술적 의미가 궁금한 사람`

이 네 경로만 남긴다.

### 6.3 Core

Core는 현재보다 더 강하게 `핵심 아이디어 소개 페이지`가 되어야 한다.

핵심 질문은 아래 3개만 우선 노출한다.

- SALT는 무엇을 새롭게 보나
- 왜 기존 물리학의 설명 공백을 하나의 흐름으로 읽으려 하나
- 이 아이디어가 검증 가능한 이유는 무엇인가

장별 압축, 세부 개념, 논리 지도는 아래쪽으로 내린다.

### 6.4 Verification

Verification은 `결과판`이 아니라 `어디까지 검토했는가`를 보여주는 페이지가 되어야 한다.

첫 화면에서 보여줄 것:

- 무엇을 비교했는지
- 현재까지 어떤 결과가 나왔는지
- 이 결과가 최종 확정이 아니라 공개 비교 결과라는 점

뒤로 뺄 것:

- 채널별 세부 링크 나열
- 후보 가설 목록 과다 노출
- 감사용 표현

### 6.5 Reference

Reference는 자료실이 아니라 `책으로 들어가는 문` 역할을 해야 한다.

첫 화면에서 보여줄 것:

- 책의 어느 장이 웹의 어느 설명과 연결되는가
- 용어가 헷갈릴 때 어디를 보면 되는가
- 더 자세한 논증은 책에서 본다는 점

### 6.6 Audit / Snapshots / Runs

이 계층은 유지하되 노출 우선순위를 낮춘다.

원칙:

- 공개 유지
- 첫 화면 전면 노출 최소화
- `검증과 근거` 안의 고급 확인 자료로 배치

즉, 독자는 먼저 SALT를 이해하고, 필요할 때만 감사 자료로 내려가게 만든다.

## 7. 웹 카피 원칙

현재 웹 카피는 정확하지만 다소 무겁다. 단순화 이후에는 아래 원칙을 따른다.

### 7.1 먼저 직관, 그 다음 정의

나쁜 예:

- `SALT는 검증 가능성과 재현 가능성을 가진 통일장적 대안 해석 구조다`

더 나은 예:

- `SALT는 우주와 미시 세계를 따로 보지 않고, 하나의 상태 변화 문제로 다시 읽으려는 시도다.`

### 7.2 명사 나열보다 문장

- `Verification / Audit / Reference / Provenance`
보다는
- `무엇을 비교했고, 그 근거가 어디에 있는지`
처럼 문장형 카피를 쓴다.

### 7.3 숫자는 설명형으로

- `68.3%`
보다는
- `82개 비교 항목 중 56개에서 SALT가 더 작은 오차를 보였다`

### 7.4 독자가 다음 행동을 알 수 있게

각 섹션 끝은 아래 중 하나로 마무리한다.

- 더 쉬운 설명은 여기서 본다
- 검증 결과는 여기서 본다
- 자세한 논증은 책에서 본다

## 8. 단계별 실행안

### Phase 1. 목적 문장 재정렬

- 홈 hero 문구 재작성
- Guide 첫 문구 재작성
- Core 첫 문구 재작성
- Verification 첫 문구 재작성

완료 기준:

- 첫 화면 10초 안에 `SALT가 무엇인지`를 말할 수 있다

### Phase 2. 전역 내비 단순화

- 상단 라벨을 입문형으로 재정렬
- Audit/Reference 노출 강도 조절
- 메뉴 수와 의미 중복 점검

완료 기준:

- 내비만 봐도 사이트가 `입문 -> 핵심 -> 근거 -> 책` 구조임을 이해할 수 있다

### Phase 3. 페이지 첫 화면 재설계

- 홈에서 링크 허브 제거
- Core에서 장 요약보다 핵심 질문 우선
- Verification에서 채널 링크보다 비교 의미 우선
- Reference에서 자료실 느낌 축소

완료 기준:

- 페이지 첫 화면이 `내용`을 먼저 말한다

### Phase 4. 고급 자료 후순위 배치

- Audit, Runs, Snapshots를 `근거 더 보기` 계층으로 정리
- 첫 진입 흐름에서 provenance 과잉 노출 제거

완료 기준:

- 일반 독자는 압도되지 않고, 필요한 사람만 상세 근거를 볼 수 있다

### Phase 5. 책 중심 유도 강화

- 책 구매와 책 구조 안내 강화
- `웹은 입문`, `책은 깊이`라는 역할 분리 명확화

완료 기준:

- 웹사이트가 책을 대체하려 하지 않고, 책의 진입문 역할을 분명히 한다

## 9. 기존 구조와의 매핑

| 현재 자산 | 유지 여부 | 조정 방향 |
| --- | --- | --- |
| `Guide` | 유지 | 더 짧고 단순한 경로 페이지로 축소 |
| `Core` | 유지 | 장 허브보다 핵심 아이디어 허브로 재정의 |
| `Verification` | 유지 | 채널 허브보다 비교 의미 중심으로 재작성 |
| `Engineering` | 부분 유지 | 전면 노출 축소, `왜 중요한가` 보조 축으로 재편 |
| `Reference` | 유지 | 책 진입용 참고 허브로 재설계 |
| `Audit` | 유지 | 고급 자료 허브로 후순위 배치 |
| `Runs / Snapshots` | 유지 | audit 하위의 provenance 계층으로 유지 |

## 10. 성공 기준

리엔지니어링 이후 아래 질문에 `예`라고 답할 수 있어야 한다.

- 처음 온 방문자가 10초 안에 SALT를 한 문장으로 설명할 수 있는가
- 사이트 첫 화면이 구조보다 메시지를 먼저 전달하는가
- 페이지 하나당 메시지가 하나로 유지되는가
- 검증과 감사 자료가 일반 독자를 압도하지 않는가
- 웹이 책의 대체물이 아니라 책의 입문문 역할을 하는가

## 11. 최종 판단

현재 사이트는 잘못 만들어진 것이 아니라 `너무 많이 설명하려는 상태`에 가깝다.

따라서 해결 방향은 기능 삭제가 아니라 아래처럼 요약된다.

- `전면 메시지는 더 단순하게`
- `고급 자료는 더 아래로`
- `핵심 인사이트는 더 짧게`
- `깊은 논증은 책으로`

즉, 이 리엔지니어링의 목표는 `정보량 축소` 자체가 아니라, `독자가 SALT를 직관적으로 이해하는 순서를 다시 설계하는 것`이다.
