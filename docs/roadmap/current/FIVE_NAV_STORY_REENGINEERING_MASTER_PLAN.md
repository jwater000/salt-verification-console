# Five-Nav Story Reengineering Master Plan

기준일: `2026-04-06`

현재 상태: `implemented in large part, operations pending`

이 문서는 공개 웹사이트를 아래 5개 내비게이션 기준으로 재구성하기 위한 단일 실행계획안이다.

1. `소개`
2. `핵심 아이디어`
3. `내용 검증`
4. `참고자료`
5. `게시판`

이 문서의 목적은 기존 여러 계획 문서를 실제 실행 가능한 하나의 흐름으로 묶는 데 있다.

- 최종 독자 노출 구조를 먼저 고정한다.
- 각 내비 항목에 필요한 스토리 전개를 먼저 설계한다.
- 그 다음 기존 라우트와 콘텐츠를 새 구조에 1:1 또는 1:N으로 매핑한다.
- 마지막에 구현, 커뮤니티, 감사 계층을 정리한다.

주의:

- 이 문서는 기존 마스터 문서의 구조 판단을 무시하지 않는다.
- 다만 실행 순서를 `기존 구조 보정` 중심이 아니라 `최종 독자 구조 선고정` 중심으로 다시 정리한다.

## 1. 핵심 판단

현재 가장 단순하고 안정적인 접근은 아래 순서를 따르는 것이다.

1. 먼저 최종 내비게이션 구조를 확정한다.
2. 그 구조에 맞는 이야기 전개를 설계한다.
3. 기존 콘텐츠와 라우트를 그 구조에 매핑한다.
4. 중복, 과밀, 후순위 자료를 정리한다.
5. 마지막에 댓글, 게시판, 인증, DB 같은 커뮤니티 계층을 연결한다.

즉 이번 리엔지니어링은 `기존 페이지를 조금씩 손보는 작업`보다 `최종 독자 구조를 먼저 선언하고, 기존 자산을 그 구조에 재배치하는 작업`으로 보는 편이 맞다.

## 2. 목표 구조

### 2.1 공개 내비게이션

공개 웹사이트의 전면 내비게이션은 아래 다섯 항목으로 고정한다.

1. `소개`
2. `핵심 아이디어`
3. `내용 검증`
4. `참고자료`
5. `게시판`

상태 메모:

- 공개면 카피와 내비게이션은 대부분 이 구조로 반영됐다.
- 게시판 목록/상세/UI/API도 반영됐다.
- OAuth 첫 로그인과 role grant까지 확인됐다.
- 실제 운영 작성/댓글/moderation 핵심 경로 검증도 완료됐다.
- 남은 핵심 작업은 CTA 관계와 화면 밀도 같은 UI 미세조정, 마지막 문서 정합화다.

### 2.2 이 구조를 먼저 고정하는 이유

- 독자는 라우트 구조가 아니라 읽는 순서를 먼저 본다.
- 현재 가장 큰 문제는 자료 부족보다 노출 순서의 과밀함이다.
- 내비게이션이 바뀌면 각 허브의 첫 화면 문장, CTA, 드롭다운 구조, 페이지 맵까지 함께 정렬하기 쉬워진다.
- 게시판도 처음부터 최상위 항목으로 위치를 잡아야 문맥이 흐려지지 않는다.

## 3. 독자 스토리 전개

이 사이트는 최종적으로 아래 순서로 읽혀야 한다.

### 3.1 소개

- SALT가 무엇을 보려는 시도인가
- 왜 이 사이트가 존재하는가
- 책과 웹은 어떤 역할 차이를 가지는가
- 어디서부터 읽으면 되는가

### 3.2 핵심 아이디어

- 기존 물리학 설명에서 무엇이 끊겨 보이는가
- SALT는 그 장면들을 어떤 흐름으로 다시 묶는가
- 상태변수와 논리 전환은 무엇인가
- 왜 이 아이디어가 검증 가능한 구조로 이어지는가

### 3.3 내용 검증

- 무엇을 비교하고 있는가
- 현재 어디까지 공개 검토되었는가
- 어떤 항목은 결과가 있고 어떤 항목은 아직 대기 중인가
- 필요하면 어디까지 재현과 감사를 따라갈 수 있는가

### 3.4 참고자료

- 앞에서 읽은 설명 중 어디서 막히기 쉬운가
- 책과 웹은 어떻게 대응되는가
- 용어, FAQ, 도해는 어떤 순서로 보면 좋은가
- 더 깊게 읽으려면 책과 어느 부분을 이어 보면 되는가

### 3.5 게시판

- 독자는 어디에 질문과 해석을 남기는가
- 페이지별 짧은 반응과 주제형 토론은 어떻게 구분되는가
- 운영과 moderation은 어디까지 공개되고 어디서 제한되는가

## 4. 페이지 공통 전개 템플릿

모든 상위 허브는 아래 순서를 기본으로 쓴다.

1. `한 줄 정의`
2. `이야기 리드`
3. `핵심 장면 2~4개`
4. `정리 단락 또는 요약 블록`
5. `근거 / 검증 / 다음 이동`

운영 원칙:

- `요약`, `근거`, `검증식 표현`은 전면 hero보다 후반 블록에 둔다.
- 첫 화면에서 카드 나열보다 문장 중심 리드를 우선한다.
- 수치는 사라지는 것이 아니라 설명을 읽은 뒤 확인하는 위치로 이동한다.
- CTA는 `다음 읽기 -> 보조 이해 -> 검증 확인 -> 감사 자료` 순으로 둔다.

## 5. 새 내비게이션별 역할과 기존 자산 매핑

### 5.1 소개

역할:

- 사이트의 첫 진입점
- SALT 한 줄 정의
- 읽기 순서 제안
- 책과 웹의 역할 차이 안내

주요 기존 자산:

- `/`
- `/guide`
- 일부 `BookstoreLinks`
- 일부 `Book Map` 입문 설명

반영 방식:

- 홈과 Guide를 `소개` 축으로 묶는다.
- 현재의 구조 설명 카드를 줄이고, 이야기 리드와 추천 순서를 앞세운다.
- 구매 CTA는 남기되, 검증 CTA보다 앞세우지 않는다.

### 5.2 핵심 아이디어

역할:

- SALT의 중심 논리를 설명하는 주축
- 문제 제기에서 개념 전환까지 보여주는 허브

주요 기존 자산:

- `/core`
- `/core/logic-map`
- `/core/chapters`
- `/core/chapters/[chapter]`
- 일부 `/engineering`

반영 방식:

- `Core`와 `Logic Map`을 사실상 같은 이야기 축으로 묶는다.
- 장 번호 중심 허브보다 `Problem -> Clue -> Concept -> Solution -> Bridge to Verification` 흐름을 먼저 보여준다.
- `Engineering`은 이 섹션의 하위 또는 보조 페이지로 읽히게 한다.

### 5.3 내용 검증

역할:

- 공개 비교 결과
- 검증 채널 구조
- 대기 항목
- 감사와 provenance로의 연결

주요 기존 자산:

- `/verification`
- `/verification/results`
- `/verification/pending`
- `/verification/channels`
- `/verification/channels/[channel]`
- `/verification/candidate-hypotheses`
- `/audit`
- `/audit/reproduce`
- `/audit/datasets`
- `/audit/formulas`
- `/audit/sources`
- `/audit/runs`
- `/runs`
- `/runs/[runId]`
- `/snapshots`
- `/snapshots/[snapshotId]`

반영 방식:

- `Verification`을 전면 허브로 유지하되, 독자에게는 `내용 검증`이라는 의미로 보이게 한다.
- `Audit`는 독립 주 메시지가 아니라 이 섹션 내부의 고급 하위 계층으로 내린다.
- `Runs`와 `Snapshots`는 후면 근거 자료로 유지한다.

### 5.4 참고자료

역할:

- 이해를 안정화하는 중간층
- 책-웹 대응
- 용어, FAQ, 도해 정리

주요 기존 자산:

- `/reference`
- `/reference/book-map`
- `/reference/faq`
- `/reference/glossary`
- `/reference/visual-atlas`

반영 방식:

- 자료실 느낌보다 `이해 보조` 역할을 앞세운다.
- Book Map을 구조 대응의 중심 축으로 둔다.
- 도해와 FAQ는 앞에서 읽은 설명을 보완하는 위치에 놓는다.

### 5.5 게시판

역할:

- 방문자 질문과 해석을 모으는 공개 대화 공간
- 페이지별 댓글과 별도의 주제형 토론 계층

주요 기존 자산:

- 페이지별 `CommentsPanel`
- `/audit/comments`
- `board_posts`, `post_comments`, `user_roles` 등 스키마
- OAuth / Neon / Prisma 준비 코드

반영 방식:

- 페이지 하단 댓글은 유지한다.
- 별도 공개 게시판 허브를 둔다.
- 운영자 관리와 moderation은 `Audit` 또는 관리자 하위에 둔다.
- 게시판은 전면에 두되, 이론·검증 허브를 대체하지 않게 한다.

## 7. 반드시 분리해서 봐야 하는 것

### 7.1 바꿔야 하는 것

- 전면 내비게이션의 의미
- 각 허브 첫 화면의 카피
- 카드와 요약의 배치 순서
- CTA 우선순위
- 독자가 보는 이야기 흐름

### 7.2 당장 바꾸지 않아도 되는 것

- 내부 라우트 경로명
- 모든 파일 경로와 폴더 구조
- 감사용 데이터 계층의 존재 자체
- 기존 도해/데이터/검증 산출물

즉 이번 리엔지니어링은 먼저 `보이는 의미 체계`를 바꾸고, 내부 구조는 가능한 한 그에 맞춰 점진적으로 정리하는 접근이 맞다.

## 8. 실행 순서

### Phase 1. 최종 내비게이션 선고정

- `소개 / 핵심 아이디어 / 내용 검증 / 참고자료 / 게시판` 확정
- 각 항목의 역할 문장 확정
- 내부 라우트와의 매핑표 확정

완료 기준:

- 더 이상 상위 IA 명칭을 두고 흔들리지 않는다

### Phase 2. 스토리 전개 와이어프레임 작성

- 5개 내비 각각에 대해 텍스트 와이어프레임 작성
- Hero, Narrative, Summary, Evidence, CTA 순서 확정
- 요약/근거/검증식 표현의 후반 배치 규칙 확정

완료 기준:

- 각 허브가 첫 화면만 봐도 무엇을 이야기하는지 알 수 있다

### Phase 3. 기존 콘텐츠 1차 매핑

- 각 페이지와 데이터 자산을 새 내비 항목에 매핑
- `유지 / 통합 / 후순위 이동 / 삭제 후보` 분류
- Audit와 provenance의 후면 이동 범위 확정

완료 기준:

- 기존 자산이 어디로 가는지 표로 설명 가능하다

### Phase 4. 전역 내비와 공통 쉘 개편

- 상단 내비 라벨 변경
- 드롭다운 설명 문장 변경
- 공통 섹션 헤더, 허브 소개 블록, CTA 규칙 반영

완료 기준:

- 내비만 봐도 읽는 순서가 보인다

### Phase 5. 허브 페이지 리엔지니어링

우선순위:

1. `소개`
2. `핵심 아이디어`
3. `내용 검증`
4. `참고자료`
5. `게시판`

완료 기준:

- 상위 허브 전부가 같은 전개 논리를 따른다

### Phase 6. 세부 페이지 재배치

- 장별 인덱스
- 채널 인덱스
- Book Map
- FAQ
- Audit 하위 페이지
- Discussion / Board 상세

완료 기준:

- 세부 페이지도 상위 허브의 역할을 거스르지 않는다

### Phase 7. 커뮤니티 스택 연결

- OAuth env 실제 연결
- Neon/Postgres migration 배포
- 게시판 읽기/쓰기 정책 확정
- moderation과 운영자 화면 연결

완료 기준:

- 게시판과 댓글이 실제 DB와 인증을 기반으로 일관되게 보인다

## 9. 커뮤니티 관련 현재 진단

현재 기준으로는 아래를 명확히 구분해야 한다.

- 페이지별 댓글 계층은 코드와 UI가 존재한다
- moderation 화면도 존재한다
- 공개 게시판 UI와 상세/작성 흐름도 이미 붙어 있다
- 운영 환경에서 `DATABASE_URL`, `AUTH_SECRET`, OAuth env가 빠지면 여전히 DB 연결과 로그인은 열리지 않는다

즉 현재 남은 문제는 `게시판이 없다`가 아니라 `운영 검증을 끝까지 닫지 않았다`에 가깝다.

1. 페이지 댓글과 게시판의 CTA 관계가 아직 완전히 정리되지 않았다
2. 신고, 숨김, 삭제, moderation을 실제 데이터 기준으로 한 바퀴 끝내지 않았다
3. 운영 검증 결과가 roadmap/ops 문서에 최종 반영되지 않았다

따라서 남은 순서는 `운영 시나리오 검증 -> 문서 체크 상태 반영`이 맞다.

## 11. 최종 판단

앞으로는 `기존 문서 3개를 동시에 보며 타협하는 방식`보다 아래 방식으로 진행하는 것이 더 낫다.

1. 먼저 최종 독자 내비 구조를 선언한다
2. 그 구조에 맞는 이야기 전개를 설계한다
3. 기존 콘텐츠를 그 구조에 매핑한다
4. 필요한 것만 구현과 데이터 계층으로 내린다

즉 이 계획의 핵심은 `기존 구조를 조금씩 수정하는 것`이 아니라 `최종 독자 구조를 먼저 고정하고, 나머지를 거기에 맞춰 정리하는 것`이다.
